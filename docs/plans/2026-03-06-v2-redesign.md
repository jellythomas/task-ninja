# Task Ninja v2 Redesign

**Date:** 2026-03-06
**Status:** Brainstorm Finalized

---

## 1. Project-Based Board (Option A)

### Current State
- Single `project_path` per run
- One flat kanban board for all tickets
- User prompted for directory each time

### Target State
- **Board = Repository** — each registered repo gets its own tab/board
- Tabs at the top of the UI for switching between repos
- Within each tab: full kanban board with all tickets targeting that repo
- Epic filter dropdown within each board to narrow by epic
- Multiple epics can coexist on the same board

### Repo Registration
- Settings page to **register repositories** (name, local path, default branch)
- Add / Edit / **Soft-delete** (if tickets are associated, archive instead of hard delete)
- Repos available as dropdown when assigning tickets
- Stored in SQLite (new `repositories` table)

### Jira Label-to-Repo Mapping
- Configurable mapping in settings: Jira label/component → registered repo
- Examples: `[MC]` → mekari_credit, `[CKYC]` → kyc_service_backend, `[BenefitCMS]` → benefit_cms
- When loading epic children, auto-assign tickets to repos based on this mapping
- Manual override always available per ticket (dropdown)
- If no mapping matches, ticket lands in an "unassigned" state requiring manual assignment

## 2. Configurable Parent Branch

### Current State
- Worktrees always branch from HEAD (current branch)
- No parent branch configuration

### Target State
- **Three levels of parent branch configuration:**

| Level | Example | Precedence |
|-------|---------|------------|
| Repo default | `main` or `master` (set during repo registration) | Lowest |
| Per-run / per-epic | `epic/MC-500` (set when creating/loading a run) | Medium |
| Per-ticket | `feat/kyc-v2` (override on individual ticket) | Highest |

- GitManager updated: `git worktree add -b feat/MC-501 .worktrees/worktree-mc-501 <parent_branch>`
- UI: parent branch field auto-filled with default, editable at epic and ticket level
- Validation: check that parent branch exists locally or in remote before queueing

## 4. Terminal Access from Tasks

### Current State
- Read-only SSE log stream in terminal panel
- No interactive shell access

### Target State

#### A) In-Browser Interactive Terminal
- WebSocket-based shell session connected to the ticket's worktree directory
- Uses xterm.js (already included) + node-pty or equivalent Python PTY library
- Opens as a panel/overlay from ticket card
- Session lifecycle tied to worktree existence
- Multiple concurrent terminal sessions supported (one per ticket)

#### B) Open in External Terminal
- "Open in Terminal" button on ticket card
- macOS: `open -a Terminal <worktree_path>` (or iTerm2 if available)
- Linux: `xdg-open` or configurable terminal emulator command
- Configurable external terminal command in settings

## 5. Replace Bitbucket MCP with Direct Jira API Calls

### Current State
- `claude_helper.py` shells out to `claude --print` to invoke MCP tools
- Burns tokens for simple Jira queries (search, transitions)
- Depends on `mcp-atlassian-with-bitbucket` MCP server being configured

### Target State

#### Scope — Orchestrator Only
- **Move to direct REST API:** Jira operations (search issues, get issue, get transitions, transition issue)
- **Keep in worker:** PR creation (via Claude CLI's `/open-pr` — already runs in worktree context)
- **No change needed:** Branch creation (local git operations)

#### Jira Credentials
- New section in Settings UI for Jira API credentials:
  - Jira Base URL (e.g., `https://mekari.atlassian.net`)
  - API Token
  - User Email
- Stored securely (encrypted at rest in SQLite or config file)
- Credentials validated on save (test API call)

#### Setup Modal
- On first visit or if credentials are missing: show a setup modal
- Modal guides user through entering Jira credentials
- Cannot dismiss without configuring or explicitly skipping
- Clear error messaging if API connection fails

#### Implementation
- New `engine/jira_client.py` — async HTTP client (httpx/aiohttp) for Jira REST API v3
- Replaces all `claude_helper.py` Jira calls
- `claude_helper.py` reduced to only PR-related operations (or removed entirely if PR stays in worker)

## 6. Configurable Agent Profiles

### Current State
- Hardcoded to `claude --print --verbose --output-format stream-json /execute-jira-task {JIRA_KEY}`
- Only works with Claude Code CLI

### Target State

#### Agent Profiles in Settings
```
┌────────────────────────────────────────────────────────────────┐
│ Agent Profiles                                                 │
├────────────────────────────────────────────────────────────────┤
│ Name: Claude Code                                   [Default]  │
│ Command: claude                                                │
│ Args Template: --print --verbose --output-format stream-json   │
│               --dangerously-skip-permissions                   │
│               /execute-jira-task {JIRA_KEY}                    │
│ Log Format: [stream-json ▼]                                    │
│                                                                │
│ Name: Gemini CLI                                               │
│ Command: gemini                                                │
│ Args Template: -p "implement jira ticket {JIRA_KEY}"           │
│ Log Format: [plain-text ▼]                                     │
│                                                                │
│ [+ Add Profile]                                                │
└────────────────────────────────────────────────────────────────┘
```

#### Key Design Decisions
- **Args template is a single field** — all flags including skip-permissions go here, no separate fields
- **Log format is a select/dropdown** with predefined options: `stream-json`, `plain-text`
- **Claude Code profile ships as default** with pre-filled values (current config)
- **Template variables available:**
  - `{JIRA_KEY}` — e.g., MC-9174
  - `{JIRA_SUMMARY}` — ticket summary text
  - `{BRANCH_NAME}` — e.g., feat/MC-9174
  - `{WORKTREE_PATH}` — full path to worktree
  - `{PARENT_BRANCH}` — parent branch name
  - `{PROJECT_PATH}` — repo root path
- **Profile assignment:** default profile set globally, overridable per-project and per-ticket
- **Switchable:** can change a ticket's agent profile before it starts processing

#### Storage
- New `agent_profiles` table in SQLite
- `repositories` table gets `default_profile_id` FK
- `tickets` table gets optional `profile_id` FK override

## 8. Data Model Changes Summary

### New Tables

```sql
-- Registered repositories
CREATE TABLE repositories (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    name TEXT NOT NULL,              -- display name (e.g., "mekari_credit")
    path TEXT NOT NULL,              -- absolute local path
    default_branch TEXT DEFAULT 'main',
    default_profile_id INTEGER REFERENCES agent_profiles(id),
    is_deleted INTEGER DEFAULT 0,   -- soft delete
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- Jira label-to-repo mapping
CREATE TABLE label_repo_mappings (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    jira_label TEXT NOT NULL,        -- e.g., "[MC]", "[CKYC]"
    repository_id INTEGER NOT NULL REFERENCES repositories(id),
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- Agent profiles
CREATE TABLE agent_profiles (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    name TEXT NOT NULL,              -- e.g., "Claude Code"
    command TEXT NOT NULL,           -- e.g., "claude"
    args_template TEXT NOT NULL,     -- e.g., "--print ... {JIRA_KEY}"
    log_format TEXT DEFAULT 'plain-text',  -- "stream-json" | "plain-text"
    is_default INTEGER DEFAULT 0,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- Jira API credentials
CREATE TABLE settings (
    key TEXT PRIMARY KEY,
    value TEXT NOT NULL,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

### Modified Tables

```sql
-- tickets: add repo, branch, and profile references
ALTER TABLE tickets ADD COLUMN repository_id INTEGER REFERENCES repositories(id);
ALTER TABLE tickets ADD COLUMN parent_branch TEXT;         -- per-ticket override
ALTER TABLE tickets ADD COLUMN profile_id INTEGER REFERENCES agent_profiles(id);

-- runs: add parent branch override (per-epic level)
ALTER TABLE runs ADD COLUMN parent_branch TEXT;            -- per-run/epic override
ALTER TABLE runs ADD COLUMN repository_id INTEGER REFERENCES repositories(id);
```

### State Enum Update
- Rename `PENDING` → `TODO` across: models/ticket.py, engine/state.py, engine/orchestrator.py, static/index.html

## 9. API Endpoints (New/Modified)

### Repository Management
- `GET    /api/repositories` — list all (exclude soft-deleted)
- `POST   /api/repositories` — register new repo
- `PUT    /api/repositories/{id}` — update repo config
- `DELETE /api/repositories/{id}` — soft-delete

### Label Mapping
- `GET    /api/label-mappings` — list all mappings
- `POST   /api/label-mappings` — add mapping
- `DELETE /api/label-mappings/{id}` — remove mapping

### Agent Profiles
- `GET    /api/profiles` — list all profiles
- `POST   /api/profiles` — create profile
- `PUT    /api/profiles/{id}` — update profile
- `PUT    /api/profiles/{id}/default` — set as default
- `DELETE /api/profiles/{id}` — delete profile

### Settings / Credentials
- `GET    /api/settings` — get all settings (credentials masked)
- `PUT    /api/settings` — update settings (Jira URL, token, email)
- `POST   /api/settings/test-jira` — test Jira API connection

### Jira (Direct API — replaces claude_helper)
- `GET    /api/jira/epic/{key}/children` — fetch epic children
- `POST   /api/jira/issue/{key}/transition` — transition issue status

### Terminal WebSocket
- `WS     /ws/terminal/{ticket_id}` — interactive shell session in worktree

### Modified Existing
- `POST   /api/tickets` — add `repository_id`, `parent_branch`, `profile_id` fields
- `PUT    /api/tickets/{id}` — allow updating repo, branch, profile assignment
- `POST   /api/runs` — add `parent_branch` field
